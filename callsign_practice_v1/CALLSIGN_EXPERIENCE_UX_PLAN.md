# 航空呼号听写练习网站体验优化方案

## 文档目标

这份方案基于当前项目的实际状态撰写：

- 前端是单页 `Astro + 原生 JavaScript`
- 当前核心流程是 `生成题目 -> 播放音频 -> 作答 -> 反馈 -> 下一题`
- 已有语速、口音、无线电音色、输入题 / 4 选 1 等能力

目标不是推倒重做，而是在现有页面和状态流基础上，把产品从“可用工具”推进到“有节奏、有反馈、有连续练习动力的学习产品”。

## 1. 体验设计总思路

### 1.1 最适合这个产品的心理学机制

这个产品最适合采用以下几类机制：

1. 即时反馈
   用户刚做完题时，是学习动机最强的瞬间。反馈越快、越明确、越带情绪，越能强化“我正在学会”。

2. 小步推进
   听写训练天然容易疲劳，不能靠长篇说明驱动，而要靠“再来一题”这种低决策成本推进。

3. 连续强化
   连对、连续完成、连续听懂，会比单次正确更容易形成投入感。用户需要看到“我不是碰巧答对，而是在稳定变强”。

4. 可感知进步
   对听觉训练来说，进步本身不容易被看见，所以界面要主动把进步翻译出来，比如“连续答对 4 题”“本轮正确率 80%”“你刚刚连续识别了 3 个 GA suffix”。

5. 轻量奖励而非重度惩罚
   航空训练本身已经带一点严肃感，如果再叠加强惩罚，很容易让体验变紧。更适合做“做对有奖励、做错有引导”，而不是“做错就打断节奏”。

### 1.2 Duolingo 式体验中适合迁移的点

适合迁移：

1. 单题节奏清晰
   每一题都要有明确的开始、作答、结算、推进，不让页面处于“我现在该做什么”的模糊状态。

2. 强状态反馈
   播放中、等待输入、已提交、正确、错误、可进入下一题，这些状态都应该有可见差异。

3. 奖励拆小
   不要等到练习结束才给成就感。单题答对、连对达到阈值、完成 5 题，都应该有轻量奖励。

4. 继续按钮的节奏控制
   “正确/错误反馈”出现后，不应瞬间跳题，而应给用户一个很短的吸收时间，再明确引导进入下一题。

5. 微动效增强情绪
   对了要有“顶一下”的上扬感，错了要有“顿一下”的收缩感，节奏差异会直接塑造情绪体验。

### 1.3 不适合直接照搬的点

不适合直接照搬：

1. 过度卡通化视觉
   Duolingo 的卡通奖励适合泛教育产品，但航空训练更适合“专业 + 游戏化”的混合气质。建议走“飞行训练界面 + 游戏反馈”而不是儿童教育感。

2. 过多剧情式角色引导
   当前产品是高频短练习工具，角色对话会稀释听觉专注。更适合用短状态文案和成就提示，不适合做长对话式教练。

3. 强生命值惩罚
   听写训练错误很常见，尤其口音切换和无线电音色开启后，若做成“错几题就失败”，会放大挫败感，不利于持续练习。

4. 复杂世界地图式关卡
   当前技术栈和产品阶段更适合“单轮 session”的轻量游戏化，而不是重导航、重内容编排的关卡系统。

## 2. 当前产品的用户流程优化

核心原则：把一次练习从“操作页面”改成“回合体验”。

### 2.1 进入练习

当前问题：

- 设置项和答题区同权重，进入成本高
- 首次用户不容易迅速明白最佳起步方式

优化建议：

1. 默认进入“快速开始”状态
   页面首屏默认给一个推荐配置，例如：
   `General Aviation + Type answer + Mixed English + 1.2x + Radio tone on`

2. 将设置区收敛成“练习配置抽屉”
   首屏只显示最关键的 2 到 3 项：
   - 练习模式
   - 作答模式
   - 开始训练按钮
   其他项如口音、语速、无线电音色放到“高级设置”折叠区。

3. 首次启动提示改成一句话 onboarding
   示例：
   `先听，再答。答完马上知道是否听对。`

为什么有效：

- 降低首次决策负担
- 用户更快进入第一题
- 训练产品最重要的是尽快进入行为，而不是先研究控件

### 2.2 听题

当前问题：

- 播放动作偏工具化，像普通音频控件
- “音频正在播放”没有成为主导状态

优化建议：

1. 把“播放题目”升级为本轮主按钮
   文案建议根据状态变化：
   - 初始：`Play Callsign`
   - 播放中：`Listening...`
   - 播放后：`Play Again`

2. 增加播放中的专属视觉态
   播放时，题目卡片进入 active 状态：
   - 外圈声波脉冲
   - 飞行通信风格频谱线
   - 主按钮轻微呼吸动画

3. 播放结束后，焦点自动落到作答区
   输入模式：自动 focus 输入框
   选择题模式：第一个选项进入键盘 focus

为什么有效：

- 把用户注意力从“界面”拉回“听觉任务”
- 强化“每题先听”的行为习惯
- 让音频成为练习节奏起点

### 2.3 作答

当前问题：

- 输入和选择都偏静态
- 提交动作情绪弱

优化建议：

1. 输入模式给出“待提交感”
   输入框内字符一旦非空，提交按钮从弱态切为强态，提示用户“可以结算了”。

2. 选择题模式加入单选预确认态
   点击选项后先进入 selected 态，再自动或点击提交结算。
   如果想更像 Duolingo，推荐：
   - 点击即选中
   - 底部主按钮变成 `Check`
   - 再点 `Check` 结算
   这样节奏更完整，也更有期待感。

3. 作答区尽量只保留一件事
   用户作答时，弱化其他设置和说明，避免视觉分心。

为什么有效：

- 强化“我正在答这一题”的专注感
- 让提交变成清晰动作，而不是顺手点击

### 2.4 查看反馈

当前问题：

- 正误反馈现在是纯文本，情绪价值不足
- 做错后没有给出学习型解释层次

优化建议：

1. 反馈分两层
   第一层是情绪反馈：
   - 正确：`Correct`
   - 错误：`Not quite`

2. 第二层是学习反馈
   - 正确时：显示正确答案 + 连对信息
   - 错误时：显示正确答案 + 可再听一次 + 可看展开答案

3. 反馈结束后出现明确推进动作
   底部主按钮切换为：
   - 正确：`Next Callsign`
   - 错误：`Try Another`

为什么有效：

- 先给情绪结论，再给学习信息，认知负担更低
- 让反馈既能鼓励，也能帮助改正

### 2.5 进入下一题

当前问题：

- 当前“Generate and Play”更像重新触发工具动作，而不是进入下一回合

优化建议：

1. 结算后主按钮统一切换为下一题动作
   用户不需要重新理解按钮。

2. 下一题切换做成短回合切换
   - 当前反馈条收起
   - 题目卡片轻微上移淡出
   - 新题淡入并自动播放

3. 每 5 题插入一次轻量里程碑
   例如：
   `5 callsigns completed`
   `Accuracy 80% this round`

为什么有效：

- 把“下一题”从功能按钮，变成回合推进
- 用户更容易形成连续作答惯性

## 3. 单题页面的界面结构优化

### 3.1 推荐布局

建议把当前页面重构为 4 个层级：

1. 顶部：Session 状态条
   - 当前模式
   - 本轮进度
   - 连对 streak
   - 正确率

2. 中部主区：题目卡
   - 播放按钮
   - 播放状态动画
   - 简短题目指令

3. 下部作答区
   - 输入框或 4 选 1
   - 提交按钮

4. 底部反馈区
   - 正误反馈
   - 正确答案 / 学习提示
   - 下一题按钮

### 3.2 信息强弱关系

强信息：

- 播放按钮
- 当前答题区
- 正误反馈
- 下一题按钮

弱信息：

- 口音
- 语速
- 自定义字母设置
- 隐藏答案说明
- 技术性状态文案

原则：

- 练习中只强调“当前题正在发生什么”
- 配置项只在用户主动需要时出现

### 3.3 如何降低认知负担

1. 每个时刻只突出一个主动作
   - 听题前：`Play Callsign`
   - 作答时：`Check`
   - 结算后：`Next Callsign`

2. 减少持续可见的信息密度
   不要一直显示“Spoken phrase stays hidden during practice”这类辅助文案。练习中应默认安静。

3. 反馈信息用固定位置承载
   不要让不同状态在页面不同区域跳来跳去。

### 3.4 输入模式与 4 选 1 的差异化布局

输入模式：

- 输入框宽、居中、视觉上像主任务区
- 输入框支持大号 monospace 字体
- 提交按钮与输入框并列或下置
- 强调“自己回忆”的挑战感

4 选 1 模式：

- 选项卡片更大，支持双列排布
- 每个选项用飞行信息牌式卡片，而不是普通按钮
- 底部保留统一 `Check` 按钮，更有节奏

原因：

- 输入模式强调主动提取
- 选择模式强调听辨对比
- 两者虽然都叫答题，但认知动作不同，界面不应完全一致

## 4. 交互与动画设计

原则：动画不是装饰，而是把练习节奏“打点”出来。

### 4.1 全局节奏建议

推荐交互节奏：

1. 进入新题：180ms - 260ms
2. 播放中状态：循环弱动画
3. 提交瞬间：120ms - 180ms
4. 正误结算：320ms - 520ms
5. 下一题切换：220ms - 320ms

### 4.2 点击反馈

#### 通用按钮

视觉变化：

- 按下时缩放到 `0.98`
- 阴影减弱
- 背景色加深 6% 到 10%

时长建议：

- press in: `70ms`
- release: `120ms`

为什么有效：

- 给用户明确物理反馈
- 让按钮更像“会回应”的对象

### 4.3 播放按钮与音频播放状态

#### 播放前

视觉：

- 主按钮高亮
- 卡片中央有微弱待机光圈

#### 播放中

视觉：

- 播放按钮切换为 active 状态
- 题目卡边缘出现环形脉冲
- 一个简化的 waveform / radio sweep SVG 做横向流动
- 背景出现非常轻的扫描线移动

动画方式：

- 光圈 scale + opacity 循环
- SVG 路径 stroke-dashoffset 循环
- 背景 gradient position 缓慢移动

时长建议：

- 脉冲：`1400ms` 循环
- waveform：`900ms` 循环
- 背景扫描：`2400ms` 循环

#### 播放结束

视觉：

- active 状态缓出
- 输入框或选项区轻微上浮 12px 并淡入

时长建议：

- 结束过渡：`180ms`
- 作答区出现：`220ms`

### 4.4 提交时反馈

#### 输入题模式

提交前：

- 按钮从普通色切到强色
- 输入框边框高亮，表示“即将判定”

提交瞬间：

- 输入框和按钮短暂锁定
- 反馈区先出现一个 `Checking...` 过渡态

时长建议：

- 锁定反馈：`120ms`
- `Checking...` 存在 `150ms - 220ms`

为什么有效：

- 增加判定仪式感
- 避免点击后页面立刻跳结果的突兀感

#### 选择题模式

选中选项：

- 当前卡片描边增强
- 卡片轻微上浮 2px
- 底部 `Check` 按钮激活

时长建议：

- 选中态：`120ms`

提交后：

- 正确选项做向上轻弹
- 错选项做一次轻微横向 shake

时长建议：

- 正确弹跳：`280ms`
- 错误 shake：`360ms`

### 4.5 答对反馈

视觉变化：

- 反馈条变为绿色主态
- 正确答案卡或输入框出现外发光
- streak badge 数字增长并弹出
- 顶部进度条前进一步

动画方式：

- 反馈条从下向上滑入
- streak badge scale `0.92 -> 1.08 -> 1`
- 进度条宽度 transition

时长建议：

- 反馈条：`240ms`
- badge 弹跳：`320ms`
- 进度条：`260ms`

节奏顺序：

1. 提交锁定
2. 结果出现
3. streak / progress 同步增长
4. 下一题按钮出现

推荐文案：

- `Correct`
- `Nice catch`
- `Clean copy`
- `You heard that right`

说明：

- 文案可以轮换，但不要太花
- 保持专业感，避免过于幼态

### 4.6 答错反馈

视觉变化：

- 反馈条变为暖红或琥珀红，而非强烈警报红
- 输入框或误选项短暂 shake
- 正确答案在 150ms 后补充出现

动画方式：

- 错误元素横向 3 段式轻抖
- 正确答案信息淡入

时长建议：

- shake：`280ms - 360ms`
- 正确答案淡入：`180ms`

节奏顺序：

1. 提交锁定
2. `Not quite`
3. 错误动作
4. 正确答案与再听入口出现
5. 下一题按钮出现

推荐文案：

- `Not quite`
- `Close, but the suffix was DXF`
- `Correct answer: 1234`

### 4.7 下一题切换

视觉变化：

- 当前反馈区收起
- 当前题卡上移淡出
- 新题卡轻微下落淡入

动画方式：

- old card: `opacity 1 -> 0`, `translateY(-8px)`
- new card: `opacity 0 -> 1`, `translateY(10px -> 0)`

时长建议：

- 旧题离场：`160ms`
- 新题入场：`220ms`

为什么有效：

- 回合边界更清楚
- 用户会更自然地进入“下一轮”

## 5. 游戏化机制

原则：优先使用前端本地状态即可完成的机制，不依赖复杂后端。

### 5.1 Streak 连对

推荐做法：

- 连对数常驻在顶部
- 连对 3、5、8 时触发不同强度提示

示例：

- `3 streak`
- `5 clean copies`
- `8 in a row`

为什么有效：

- 连对比总分更能驱动“再做一题”
- 特别适合短时高频训练

### 5.2 连续练习奖励

建议以题量触发，而不是时间触发：

- 完成 5 题：提示 `Warm-up complete`
- 完成 10 题：提示 `Good rhythm`
- 完成 15 题：提示 `Strong session`

实现方式：

- 本地计数
- 使用 toast 或顶部横幅短提示

### 5.3 小成就提示

适合做的成就类型：

- 首次连续答对 5 题
- 首次完成 10 题 session
- 某一模式下连续正确 3 题
- 开启无线电音色后仍连续正确 3 题

注意：

- 只做轻提示，不做复杂勋章系统也能有效
- 文案应强调能力增长，而不是纯收集

### 5.4 本轮练习进度

推荐做“软 session”：

- 每轮默认目标 10 题
- 顶部显示 `4 / 10`
- 做满 10 题后给结算面板

为什么有效：

- 没有尽头的刷题容易疲劳
- 有小目标更容易坚持完成一轮

### 5.5 练习结束反馈

结算面板建议展示：

- 本轮完成题数
- 正确率
- 最长 streak
- 最常练习模式
- 一句鼓励反馈

文案示例：

- `10 callsigns completed`
- `Accuracy: 80%`
- `Best streak: 4`
- `You are getting faster at decoding radio-style speech`

### 5.6 如何让用户更愿意继续下一题

建议组合：

1. 下一题按钮始终是主视觉按钮
2. 正确后自动出现正向短反馈
3. streak 接近阈值时主动提示
   例如：
   `One more for a 5-streak`

4. session 结束时给两个出口
   - `Start another round`
   - `Change settings`

这样既保持惯性，也给用户控制感。

## 6. 多感官反馈设计

### 6.1 声音反馈

建议新增 3 类轻量 UI 音效：

1. 提交成功按键声
   很短，偏机械确认感

2. 正确音效
   2 到 3 个短音，上扬，不要过亮

3. 错误音效
   一个低短促提示音，克制，不刺耳

实现建议：

- 优先用 Web Audio API 实时合成简单提示音
- 不必引入音频素材包

为什么适合当前项目：

- 成本低
- 文件体积小
- 可与现有音频体系共存

### 6.2 动效反馈

动效强度建议分级：

强反馈：

- 答对
- 答错
- streak 升级
- session 完成

弱反馈：

- hover
- input focus
- 播放中
- 切换选项

原因：

- 如果所有动作都“很热闹”，用户很快疲劳
- 强反馈要留给真正重要的学习节点

### 6.3 移动端震动反馈

推荐使用 `navigator.vibrate()` 做非常轻的反馈：

- 正确：`20ms`
- 错误：`30ms, pause 30ms, 30ms`
- streak 达成：`18ms, pause 18ms, 18ms`

注意：

- 仅在支持设备上启用
- 提供设置开关
- 不要和音频播放同时做重震动

### 6.4 与“无线电音色”功能的协调

关键原则：

- 题目音频是训练主体
- UI 音效只是辅助层

具体建议：

1. 题目播放中，不播放任何抢注意力的 UI 音效
2. 正误提示音只在提交结算后播放
3. UI 音效音量显著低于题目音频
4. 无线电音色开启时，UI 音效不要也做“无线电失真”
   否则整个页面都像在噪声里，反馈会糊掉

最优策略：

- 题目音频保持航空感
- UI 反馈保持干净、短促、现代

## 7. Micro UX 细节建议

### 7.1 按钮状态

每个核心按钮至少有 4 态：

- default
- hover
- active
- disabled

主按钮建议：

- default：高识别主色
- hover：亮度 +3% 到 +5%
- active：缩放 `0.98`
- disabled：降低对比度和阴影

### 7.2 输入框聚焦与提交反馈

建议：

- focus 时边框变亮，底部辅线出现
- 输入合法字符后显示 ready 态
- 回车提交时输入框短暂锁定
- 错误时不要只变红，要同时给出正确答案入口

### 7.3 错误提示文案风格

文案原则：

- 不指责
- 不技术化
- 先结论，再信息

推荐风格：

- `Not quite. Correct answer: DXF`
- `Almost. Flight number was 1234`

不推荐：

- `Validation failed`
- `Incorrect input format`

### 7.4 空状态 / 初始状态 / 加载状态

初始状态：

- 不要只是“Generate a sample to start”
- 建议换成更像学习邀请的文案：
  `Ready for your first callsign?`

加载状态：

- 题目生成时显示短过渡：
  `Preparing transmission...`

音频请求中：

- 显示：
  `Tuning voice channel...`

为什么有效：

- 同样是加载，也能服务沉浸感
- 但要克制，避免为了拟物而妨碍理解

### 7.5 首次进入时如何降低学习成本

建议首轮做一个极轻新手引导：

1. 先听 callsign
2. 输入或选择你听到的内容
3. 提交后立刻查看反馈

形式建议：

- 首次访问时显示 3 步提示卡
- 一次关闭后本地存储，不重复打扰

## 8. 前端实现建议

### 8.1 基于 Astro + 原生 JS 的推荐实现方式

推荐继续沿用当前单页结构，在 `src/pages/index.astro` 内逐步扩展状态：

- `sessionState`
- `questionState`
- `streak`
- `roundProgress`
- `accuracy`
- `isPlaying`
- `isChecking`
- `resultState`

不要急着引入复杂前端框架。当前交互规模完全可以通过：

- DOM class 切换
- `data-state` 属性
- 小型事件函数

来稳定实现。

### 8.2 哪些动画适合用 CSS

适合 CSS 的：

- hover / active / disabled
- 反馈条滑入滑出
- 进度条推进
- 选项卡选中态
- 播放中的脉冲、呼吸、淡入淡出

原因：

- 实现简单
- 性能稳定
- 易于配合 `prefers-reduced-motion`

### 8.3 哪些动画适合用 Web Animations API

适合 Web Animations API 的：

- 答错 shake
- streak badge 弹跳
- 当前题离场 / 下一题入场
- 连对阈值触发的 toast 动画

原因：

- 这些动画通常是一次性触发
- WAAPI 比手写 class 定时器更清晰

### 8.4 音效和小反馈如何低成本实现

建议：

- 使用单独的 `ui-audio.js` 小模块
- 通过 Web Audio API 合成极短 beep / chime
- 首次用户交互后解锁 AudioContext

避免方案：

- 不要上大型音效库
- 不要用很多 mp3 小文件

### 8.5 如何避免性能问题

建议：

1. 动画尽量只动 `transform` 和 `opacity`
2. 不要对大面积模糊层做高频动画
3. 播放中动画控制在 2 到 3 个循环元素以内
4. 选项按钮重绘尽量通过 class 切换完成
5. 不要在每次音频播放时创建过多无清理对象

### 8.6 如何兼顾移动端

建议：

1. 主按钮固定在拇指易触区域
2. 输入框字号至少 16px，避免 iOS 自动缩放
3. 4 选 1 卡片高度足够，便于点按
4. 底部反馈区不要被软键盘遮挡

### 8.7 如何兼顾 reduced motion / 无障碍

必须支持：

- `prefers-reduced-motion`
- 键盘操作
- 可见 focus ring
- 正误状态不仅靠颜色，还要靠图标 / 文案

reduced motion 模式下：

- 保留颜色和状态变化
- 去掉弹跳、shake、脉冲循环
- 下一题切换只做淡入淡出

## 10. 推荐的页面气质方向

建议视觉方向定义为：

`Flight deck training` + `game-like rhythm`

关键词：

- 专业但不冰冷
- 沉浸但不压迫
- 有反馈，但不过度卡通

可以考虑的具体视觉语言：

- 字体上：标题更有识别度，答案区用清晰 monospace
- 颜色上：主色偏航空仪表暖橙 / 跑道灯绿 / 通信蓝灰
- 组件上：卡片更像训练面板，而不是普通表单
- 动效上：更多“信号、扫描、推进、确认”，少一些无关装饰

## 12. 总结

这类产品真正要优化的，不是“界面更好看”，而是把一次听写训练做成有回合感的体验：

- 听的时候专注
- 答的时候明确
- 对错时有情绪反馈
- 下一题时有推进冲动
- 做完一轮时能感觉自己真的在进步

对当前项目来说，最关键的不是加更多功能，而是把现有功能重新编排成更强的节奏。只要把 `播放 -> 作答 -> 反馈 -> 推进` 这条主链路打磨好，产品就会从“能练”明显提升到“想继续练”。
